| D001 | M001 | arch | Workflow abstraction model | WorkflowEngine + ExecutionPolicy two-layer abstraction | All three ADR reviewers converged: engine alone is insufficient because finalize() is not generic. | D007 | M001 | arch | Iteration freshness policy | Once expanded, iteration is frozen for that run | Source artifact changes after expansion do not retroactively add/remove instances. | D013 | M001 S01 T02 | architecture | How to group auto.ts's 58-module dependency surface into LoopDeps sub-interfaces | 10 role-based groups merging small clusters: ModelAndBudgetOps (model+budget+routing), ObservabilityOps (logging+dashboard+skills), RecoveryOps (recovery+doctor+health+resource-version), PostUnitOps (post-unit+closeout+hooks), SessionOps (bootstrap+locks+runtime+queue+captures). Post-unit hooks are lifecycle, not a separate concern. Leaf utilities (errors, atomic-write, unit-id) aren't engine-polymorphic and don't need sub-interfaces. | Yes |
| D014 | M001 S02 T02 | architecture | How to handle mid-function re-derivation sites in dispatchNextUnit() after engine wiring | Manual inline engineState construction at re-derivation sites rather than calling engine.deriveState() again | Mid-function re-derivation (milestone transition, merge reconciliation) calls the same deriveState() function directly. Routing these through engine.deriveState() would add double I/O for no behavioral difference in the dev engine, and the inline construction preserves exact existing behavior. The pattern keeps engine.deriveState() as the single entry point while acknowledging that internal re-derivations are an implementation detail of the dev workflow. | Yes — S03 may refactor if CustomWorkflowEngine needs different re-derivation behavior |
