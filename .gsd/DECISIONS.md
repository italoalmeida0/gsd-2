| D001 | M001 | arch | Workflow abstraction model | WorkflowEngine + ExecutionPolicy two-layer abstraction | All three ADR reviewers converged: engine alone is insufficient because finalize() is not generic. | D002 | M001 | arch | Dev workflow stays as code | DevWorkflowEngine is a TypeScript class, not YAML | Too much conditional logic, recovery heuristics, and milestone-specific behavior for YAML. The YAML schema serves custom workflows, not every possible workflow. | No |
| D003 | M001 | arch | LoopDeps decomposition strategy | Decompose into role-based sub-interfaces alongside engine extraction | LoopDeps is a 70-function god-interface. | D004 | M001 | scope | Existing workflow-templates coexistence | Keep /gsd start system unchanged, coexist with /gsd workflow | Workflow-templates serve single-session quick tasks (bugfix, spike, hotfix). | D008 | M001 | scope | Pause behavior for custom workflows | Pause just stops auto-mode, no engine switching | Simpler state machine. No complexity of falling back to dev engine during pause. | No |

| D010 | M001 | scope | All four verification policies in V1 | content-heuristic, shell-command, prompt-verify, human-review | Human-review (pause for user input) is needed for workflows where automated verification is insufficient. | No |
| D011 | M001 | scope | LLM-assisted workflow builder in V1 | Conversation with LLM yields YAML following schema structure | Users shouldn't need to learn YAML schema. Agent is the builder. Full experience: describe → build → run. | No |
| D012 | M001 | scope | Zero dev regression is non-negotiable | All existing tests must pass identically after interface extraction | Dev workflow is production. First slices (S01, S02) are pure refactoring validated by the existing 172-test suite. | No |
| D013 | M001 S01 T02 | architecture | How to group auto.ts's 58-module dependency surface into LoopDeps sub-interfaces | 10 role-based groups merging small clusters: ModelAndBudgetOps (model+budget+routing), ObservabilityOps (logging+dashboard+skills), RecoveryOps (recovery+doctor+health+resource-version), PostUnitOps (post-unit+closeout+hooks), SessionOps (bootstrap+locks+runtime+queue+captures). Low-level leaf utilities excluded entirely. | Fewer, larger groups reduce cross-group coupling for S02's engine implementations. Budget flows through model selection into routing history — splitting them creates artificial seams. Dashboard/notifications/logging are all output concerns. Doctor/health feeds recovery decisions. Post-unit hooks are lifecycle, not a separate concern. Leaf utilities (errors, atomic-write, unit-id) aren't engine-polymorphic and don't need sub-interfaces. | Yes |
| D014 | M001 S02 T02 | architecture | How to handle mid-function re-derivation sites in dispatchNextUnit() after engine wiring | Manual inline engineState construction at re-derivation sites rather than calling engine.deriveState() again | Mid-function re-derivation (milestone transition, merge reconciliation) calls the same deriveState() function directly. Routing these through engine.deriveState() would add double I/O for no behavioral difference in the dev engine, and the inline construction preserves exact existing behavior. The pattern keeps engine.deriveState() as the single entry point while acknowledging that internal re-derivations are an implementation detail of the dev workflow. | Yes — S03 may refactor if CustomWorkflowEngine needs different re-derivation behavior |
| D015 | M001 S03 | architecture | Engine ID format for custom workflows | Use "custom:/path/to/run" format — prefix "custom:" identifies engine type, suffix is run directory path | Engine resolution needs both the engine type (to pick the class) and the run-specific state directory (to read GRAPH.yaml). Encoding both in a single string avoids adding a second parameter to resolveEngine() and keeps the activeEngineId field in AutoSession as a simple string. The colon separator is unambiguous since absolute paths start with / on Unix. | Yes — if the format proves too rigid for future engine types |
| D016 | M001 S03 | architecture | How CustomWorkflowEngine navigates GSDState-dependent code in dispatchNextUnit() | GSDState stub with non-null activeMilestone ({ id: "custom-workflow", title: "Custom Workflow" }), phase "executing", and empty arrays for all collection fields | dispatchNextUnit() has ~300 lines of GSDState-specific code between engine.deriveState() and engine.resolveDispatch(). Rather than modifying auto.ts (which would risk regressions in the 1590-test suite), the custom engine returns a GSDState-compatible stub in EngineState.raw that navigates all guard conditions neutrally. activeMilestone must be non-null (otherwise mid===null triggers stopAuto before resolveDispatch). Phase must be "executing" (not "complete" or "blocked" which trigger early returns). | Yes — S04+ may refactor auto.ts to use engine-polymorphic checks instead of GSDState-specific ones |
| D017 | M001 S05 | architecture | How prompt-verify policy works in V1 | prompt-verify returns "pause" with the verification prompt surfaced in the reason message, matching human-review pattern | True LLM-driven verification (where the LLM evaluates output quality) requires dispatching a verification sub-step through the auto-loop, which adds coupling complexity. V1 pauses and shows the user the verification prompt. If user resumes, the step is considered verified. Full LLM verification is a candidate for post-V1. | Yes — post-V1 can implement true LLM-driven verification via synthetic sub-steps |
| D018 | M001 S05 | architecture | Where artifact paths resolve for context injection and verification | Resolve produces paths relative to runDir; context injector reads from runDir only (no basePath fallback in V1) | The run directory is the workflow's isolated namespace. The prompt template should instruct agents to write artifacts to the run directory. Adding a basePath fallback creates ambiguity about which copy is authoritative. Keep it simple for V1 — if artifacts aren't in runDir, context injection returns empty and verification returns retry. S07 CLI can ensure prompts include the run directory path. | Yes — if artifact write location proves problematic in practice |
| D019 | M001 S07 | architecture | When to substitute {{variable}} parameters in workflow step prompts | Substitute at dispatch time in resolveDispatch(), not at snapshot time in createRun(). Store CLI --param overrides in a separate PARAMS.json file in the run directory. | Substituting at snapshot time would make DEFINITION.yaml diverge from the source file, violating R007's byte-exact copyFileSync contract. Dispatch-time substitution keeps the snapshot truly immutable (raw source YAML) while still resolving params when building the actual prompt. PARAMS.json stores overrides separately, allowing the same definition to be run with different parameters. | No |
| D020 | M001 S07 | architecture | How to handle workflow resume after process restart when activeEngineId is lost from memory | On resume, scan workflow-runs/ for the most recent incomplete run and re-derive activeEngineId from it. Also store the engine ID in the run's directory state for robustness. | activeEngineId lives on the in-memory AutoSession and is lost on process restart. The resume command needs to know which run to continue. Scanning workflow-runs/ for incomplete runs (checking GRAPH.yaml for pending steps) is deterministic and requires no additional state files. This makes resume work even after a crash. | Yes — could add a persistent current-run pointer file if scan proves too slow |
| D021 | M001 S08 | architecture | How to render custom workflow progress in TUI widget | Conditional branch in existing "gsd-progress" factory widget with optional DisplayMetadata parameter through entire call chain. Dashboard overlay falls through silently to dev path on engine resolution failure. | One widget ID for both dev and custom workflows keeps the TUI surface simple. Optional parameter means all existing call sites work unchanged — zero regression risk. Silent fallthrough prevents overlay crashes from stale engine state. | Yes — if custom workflows need richer progress display than step N/M |
